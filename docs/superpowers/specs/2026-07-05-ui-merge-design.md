# QuantScout UI 合并设计文档

**设计日期**: 2026-07-05
**设计目标**: 将策略控制器和参数优化器合并为单 Streamlit 应用,实现统一前端、实时日志显示、公共配置合并

---

## 1. 需求总结

### 1.1 核心需求

| 需求项 | 详细描述 | 优先级 |
|--------|---------|--------|
| 架构合并 | 策略控制器 + 参数优化器 → 单 Streamlit 应用（多页面路由） | P0 |
| 日志前端显示 | 实时滚动日志替代终端打印，解决"假死判断"问题 | P0 |
| 公共配置合并 | Token、高级权重等配置统一，减少代码重复 | P1 |
| UI 风格统一 | 两个功能使用相同视觉风格 | P1 |
| 用户体验优化 | 进度监控、心跳动画、停止任务按钮 | P1 |

### 1.2 技术选型

| 技术决策 | 选择方案 | 理由 |
|---------|---------|------|
| 合并方式 | 单 Streamlit 应用 + 多页面路由 | 统一入口,简化部署,用户无需打开两个地址 |
| 后台任务方案 | threading 后台线程 + 定时刷新 | 兼容现有代码,实现相对简单 |
| 日志实时性 | 实时滚动日志（1-2 秒刷新） | 用户可实时监控进度,避免假死疑虑 |
| UI 布局 | 侧边栏导航 + 底部固定日志面板 | 功能切换清晰,日志不占用主内容区空间 |
| 实现路径 | 保留现有模块 + 统一入口 | 最小改动,风险可控,工作量适中 |

## 3. 核心组件设计

### 3.1 日志面板组件 (`components/log_panel.py`)

**类名**: `RealtimeLogPanel`

**核心机制**:
- 后台线程执行任务,通过 QueueHandler 将日志写入全局队列
- 前端组件定时轮询队列（每 1-2 秒）
- 日志显示在底部固定面板,可折叠

**关键功能**:
- 日志级别过滤（INFO/WARNING/ERROR）
- 自动滚动到最新行
- 最大行数限制（默认 500 行,超出清空旧日志）
- 支持导出完整日志文件

**UI 结构**:
```
┌────────────────────────────────────────────┐
│ [实时日志] ▼ (展开)                         │  ← 折叠按钮
│ ┌────────────────────────────────────────┐ │
│ │ [INFO] 10:23:45 选股开始                │ │
│ │ [INFO] 10:23:47 批次1完成 - 128支       │ │
│ │ [WARN] 10:23:50 Token连接超时,重试...   │ │
│ └────────────────────────────────────────┘ │
│ [清空日志] [导出日志] [级别: INFO▼]       │  ← 操作按钮
└────────────────────────────────────────────┘
```

### 3.2 进度监控组件 (`components/progress_monitor.py`)

**类名**: `ProgressMonitor`

**显示元素**:
- 进度条：`st.progress(0.6)`
- 百分比文字：60%
- 当前状态：正在处理第 6/10 批 (600/1000 股)
- 预计剩余时间：~2 分钟（基于历史平均速度估算）
- 心跳动画：状态文字旁显示闪烁图标 ⏳（表明线程存活）

**关键逻辑**:
- 后台线程通过 `session_state['progress']` 更新进度
- 前端定时刷新读取 session_state
- 提供"停止任务"按钮（设置 stop_flag）

**UI 结构**:
```
┌────────────────────────────────────────────┐
│ [选股进度] ████████████░░░░░░░░ 60%        │
│ 当前状态: 正在处理第 6/10 批 ⏳            │  ← 心跳动画
│ 预计剩余: ~2 分钟                          │
│ [停止选股]                                 │
└────────────────────────────────────────────┘
```

### 3.3 公共配置组件 (`components/common_config.py`)

**类名**: `CommonConfigPanel`

**合并项**:
- Token 配置（原 `strategy_controller/ui/token_component.py`）
- 高级权重配置（权重因子、重点指标选择）
- 测试模式开关

**存储方式**:
- 配置写入 `session_state['common_config']`
- 两个页面共享,切换页面时保持状态

**UI 位置**:
- 侧边栏顶部固定区域（不受页面切换影响）

**侧边栏结构**:
```
┌────────────────────────────┐
│ [公共配置]                  │
│ ├─ Token: 已配置 ✓         │
│ ├─ 测试模式: [开关]        │
│ ├─ 高级权重配置 (expander) │
│ │   ├─ 重点指标: [kdj_j]   │
│ │   └─ 权重因子: 1.5       │
│ └─────────────────────────│
│ [页面导航]                  │  ← 页面切换
│ ├─ 策略控制器              │
│ └─ 参数优化器              │
└────────────────────────────┘
```

## 4. 日志系统优化设计

### 4.1 日志优化策略

**类名**: `LogManager`

**核心目标**:
- 减少日志量：从"逐股打印"改为"批次摘要"
- 保留关键信息：错误、警告、阶段切换、进度节点
- 前端友好：结构化日志（JSON 格式）便于 UI 解析

**过滤策略对比**:
```
原日志模式 (逐股打印):
├─ [INFO] 处理股票 000001.SZSE
├─ [INFO] 处理股票 000002.SZSE
├─ [INFO] 处理股票 000003.SZSE
... (1000 行)

优化后 (批次摘要):
├─ [INFO] 批次1开始 - 100支股票
├─ [INFO] 批次1筛选完成 - 符合条件: 23支
├─ [INFO] 批次2开始 - 100支股票
├─ [INFO] 批次2筛选完成 - 符合条件: 18支
... (10 行)
├─ [INFO] 选股完成 - 总计: 128支 (耗时: 45秒)
```

### 4.2 日志级别使用规范

| 级别 | 使用场景 | 示例 |
|------|---------|------|
| INFO | 阶段开始/结束、批次摘要、关键进度节点 | `"批次1筛选完成 - 符合条件: 23支"` |
| WARNING | Token 连接异常、网络重试、非阻断问题 | `"Token连接超时,重试..."` |
| ERROR | 选股失败、回测异常、需要用户介入的问题 | `"选股失败: 网络连接中断"` |
| DEBUG | 详细调试信息（默认不显示,用户可开启） | `"股票000001详细评分数据"` |

### 4.3 结构化日志格式

```json
{
    "level": "INFO",
    "timestamp": "2026-07-05 10:23:45",
    "message": "批次1筛选完成",
    "data": {"batch": 1, "total": 100, "matched": 23},
    "type": "progress"
}
```

`type` 字段用于前端识别并渲染进度条。

### 4.4 具体优化场景

| 原场景 | 原日志量 | 优化后日志量 | 示例 |
|--------|---------|-------------|------|
| 选股 1000 股（10 批次） | ~1000 行 | ~10 行 | `"批次3完成 - 符合条件: 35支"` |
| 参数优化 500 组合 | ~5000 行 | ~50 行 | `"优化完成 - 最佳参数: 止盈12%, 止损-3%"` |
| 回测单只股票 | ~200 行 | ~5 行 | `"回测完成 - 收益率: 18.5%, 夏普比率: 1.2"` |

## 5. 数据流与线程同步设计

### 5.1 后台任务执行器

**类名**: `TaskRunner`

**核心机制**:
- 任务线程：执行选股/回测等阻塞任务
- 进度队列：任务线程写入进度/日志
- 控制队列：前端线程写入停止信号

**数据流**:
```
前端线程 (Streamlit Main Thread):
├─ 用户点击"开始选股"
├─ 创建 TaskRunner 实例
├─ 启动后台线程执行选股任务
├─ 定时轮询 progress_queue (每 1-2 秒)
│   ├─ 读取进度 → 更新 st.progress
│   ├─ 读取日志 → 更新日志面板
│   └─ 读取状态 → 更新状态文字
├─ 用户点击"停止选股"
│   └─ 写入 stop_queue → 任务线程检测并停止
└─ 结束

后台线程 (选股任务线程):
├─ 初始化：创建 progress_queue, stop_queue
├─ 循环处理每批次
│   ├─ 写入进度：queue.put({'type': 'progress', ...})
│   ├─ 写入日志：queue.put({'type': 'log', ...})
│   ├─ 检查停止信号：if stop_queue.get(timeout=0) break
│   └─ 执行选股逻辑（调用现有 strategy_executor）
├─ 任务完成：queue.put({'type': 'complete', ...})
└─ 结束
```

### 5.2 队列数据格式

```python
# 进度消息
{'type': 'progress', 'value': 0.6, 'text': '正在处理第6/10批'}

# 日志消息
{'type': 'log', 'level': 'INFO', 'message': '批次6筛选完成'}

# 完成消息
{'type': 'complete', 'success': True, 'result': {...}}

# 错误消息
{'type': 'error', 'message': '网络连接失败'}
```

### 5.3 线程同步关键点

| 同步问题 | 解决方案 |
|---------|---------|
| 任务线程写入进度 | 使用 `queue.put()`,前端线程通过 `queue.get(timeout=0)` 非阻塞读取 |
| 前端线程发送停止信号 | 使用独立的 `stop_queue`,任务线程每批次检查一次 |
| session_state 跨线程访问 | Streamlit 不支持线程间直接共享 session_state,通过队列间接传递 |
| 任务异常捕获 | 任务线程内部 `try-except`,将异常写入队列,前端线程渲染错误提示 |

## 9. 风险评估

| 风险点 | 风险等级 | 缓解措施 |
|--------|---------|---------|
| threading 与 Streamlit 兼容性 | ⭐⭐ 中 | 使用队列而非直接操作 session_state,测试多种刷新策略 |
| 日志队列阻塞 | ⭐⭐ 中 | 队列设置最大长度,超出时丢弃旧日志 |
| 现有业务逻辑改动影响稳定性 | ⭐⭐⭐ 高 | 仅改造 UI 层,业务逻辑保持不变,充分测试 |
| 任务停止机制不响应 | ⭐⭐ 中 | 增加停止检测频率（每批次检查）,提供强制终止按钮 |
