# 选股和回测效率优化方案

## 一、当前流程分析

### 1. 选股流程
- **核心文件**：`data/batch_processor.py`
- **主要方法**：`process_all_stocks_parallel`
- **流程**：
  1. 顺序处理股票批次（已修复嵌套ThreadPoolExecutor问题）
  2. 每个批次内部：
     - 批量获取股票基础信息
     - PE/市值筛选（内存中快速筛选）
     - **并行KDJ计算**（使用ThreadPoolExecutor）
     - **并行趋势分析和评分**（使用ThreadPoolExecutor）

### 2. 回测流程
- **核心文件**：`ulti-para-seeker/optimizers/brute_force_optimizer.py`
- **主要方法**：`run_parallel_optimization`
- **问题**：虽然方法名包含"parallel"，但实际上是**顺序处理**每个参数组合
- **每个参数组合的回测步骤**：
  1. 创建策略ID和回测区间
  2. 生成配置文件
  3. 调用`backtest_runner.run_backtest`
  4. 读取回测结果文件

### 3. 缓存和数据库读取流程
- **核心文件**：`cache/data_cache.py`
- **主要类**：`StockDataCache`
- **数据库**：SQLite
- **缓存策略**：
  - K线数据：缓存到SQLite，支持增量更新
  - 基础信息：缓存到SQLite
  - 回测数据：永久化存储

## 二、效率瓶颈分析

### 1. 回测流程未真正并行
- 每个参数组合顺序执行，未充分利用多核CPU
- 频繁的IO操作：每个回测都需要读写配置文件和结果文件

### 2. 选股流程效率问题
- 批次处理是顺序的：虽然批次内部并行，但批次之间是串行处理
- 日志输出过于频繁：影响性能
- 重复初始化和数据加载：每次回测都重新加载策略参数和数据

### 3. 缓存和数据库读取环节（重点分析）
从日志中可以看到，缓存检查和数据获取环节耗时较长，主要问题包括：

#### 3.1 数据库连接管理问题
- **频繁的数据库连接创建和销毁**：每次缓存操作都创建新的SQLite连接，带来很大开销
- **单一线程锁**：整个缓存实例使用一个线程锁，导致并发性能下降
- **未使用连接池**：没有连接池管理，每次操作都需要重新建立连接

#### 3.2 缓存操作效率问题
- **同步的缓存写入**：每次缓存写入都同步执行，阻塞主线程
- **复杂的缓存命中逻辑**：在`get_cached_kline_data`中，有复杂的日期检查逻辑，影响性能
- **冗余的缓存键生成**：使用MD5生成缓存键，增加CPU开销
- **JSON序列化/反序列化开销**：大量数据的JSON转换消耗CPU资源

#### 3.3 批量操作效率问题
- **逐个检查缓存**：在`get_stock_basic_info_batch`中，逐个检查缓存，效率低下
- **批量API调用失败回退**：批量API调用失败时，回退到逐个调用，效率低下
- **频繁的日志输出**：在缓存操作中频繁输出日志，影响性能

#### 3.4 缓存策略问题
- **缓存命中率低**：虽然缓存检查完成，但可能有大量缓存未命中
- **异步更新最后访问时间**：创建大量线程，增加系统开销
- **数据库索引未充分利用**：查询逻辑可能没有充分利用索引

## 三、优化方案

### 1. 真正并行的回测处理
- **实现方式**：使用`ProcessPoolExecutor`替代顺序处理
- **优势**：
  - 充分利用多核CPU，避免GIL限制
  - 每个进程处理独立的参数组合
  - 减少单线程阻塞
- **代码修改**：
  ```python
  # 在run_parallel_optimization方法中
  with ProcessPoolExecutor(max_workers=os.cpu_count()) as executor:
      futures = [executor.submit(self.run_backtest, param) for param in param_combinations]
      for future in as_completed(futures):
          result = future.result()
          all_results.append(result)
  ```

### 2. 优化选股流程
- **恢复批次并行处理**：使用单个ThreadPoolExecutor并行处理批次
- **代码修改**：
  ```python
  # 在process_all_stocks_parallel方法中
  with ThreadPoolExecutor(max_workers=effective_workers) as executor:
      # 提交所有批次任务
      futures = []
      for batch_num in range(total_batches):
          batch_symbols = all_symbols[batch_num*batch_size : (batch_num+1)*batch_size]
          future = executor.submit(self.process_batch_parallel, batch_symbols, trade_date, batch_num+1)
          futures.append(future)
      # 收集结果
      for future in as_completed(futures):
          all_results.extend(future.result())
  ```

### 3. 缓存和数据库优化（重点）

#### 3.1 数据库连接优化
- **使用连接池管理数据库连接**：减少连接创建和销毁开销
- **实现方式**：使用`sqlite3`的`check_same_thread=False`选项，并实现简单的连接池
- **代码修改**：
  ```python
  # 在StockDataCache类中
  def __init__(self, db_path="stock_data_cache.db", cache_expiry_days=7, permanent_storage_for_backtest=True):
      self.db_path = db_path
      self._lock = threading.RLock()  # 使用可重入锁
      self.cache_expiry_days = cache_expiry_days
      self.permanent_storage_for_backtest = permanent_storage_for_backtest
      self._init_database()
      self._create_config_table()
      # 创建连接池
      self._conn_pool = []
      self._max_connections = 10
      # 初始化连接池
      for _ in range(self._max_connections):
          self._conn_pool.append(sqlite3.connect(self.db_path, check_same_thread=False))
  ```

#### 3.2 缓存键设计优化
- **直接使用symbol+trade_date作为缓存键**：避免MD5计算和冗余查询
- **代码修改**：
  ```python
  def _generate_cache_key(self, symbol: str, trade_date: str, data_type: str, days: int = None) -> str:
      """生成缓存键"""
      key_parts = [symbol, trade_date, data_type]
      if days is not None:
          key_parts.append(str(days))
      return "_".join(key_parts)
  ```

              ''', (cache_key,))
              result = cursor.fetchone()
              self._return_connection(conn)
          
          if result:
              data_json, created_time, is_permanent = result
              
              # 只有非永久化数据才需要检查过期时间
              if not bool(is_permanent):
                  # 检查数据是否过期
                  cache_time = datetime.datetime.strptime(created_time, '%Y-%m-%d %H:%M:%S')
